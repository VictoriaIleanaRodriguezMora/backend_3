## Seguridad y documentación

### El temor de todo desarrollador…

Si has llegado hasta este curso avanzado, seguramente tienes los conocimientos para desarrollar un servidor backend desde cero mediante buenas prácticas.

Sin embargo, hay algo más de lo que hay que hablar sobre el desarrollo, además de las arquitecturas, testing y escalabilidad: **las vulnerabilidades**.

Me imagino que en más de una ocasión habrás escuchado hablar sobre el hackeo de X plataforma, sobre el robo de información de X base de datos, o la simple caída de X página web debido a alguna persona malintencionada.

Cuando nuestro servidor se encuentre arriba, siempre tendremos este pequeño cosquilleo:

- “¿Y si en algún momento, a causa de mi sistema web, alguien logra extraer información de mis usuarios?”
- “No quiero que sea mi página web la que comprometa la información sensible de mis clientes”

---

### ¿Qué es una vulnerabilidad?

En el mundo IT, una **vulnerabilidad** es cualquier tipo de debilidad o brecha dejada en nuestro sistema, que permita a una persona malintencionada aprovecharla para comprometer la seguridad del sistema o el usuario.

Existen múltiples tipos de vulnerabilidades:

- **De hardware**: cuando la implicación de seguridad se da en un elemento físico (como el servidor).
- **De software**: cuando la implicación de seguridad se da en nuestro aplicativo.
- **Procedimentales**: cuando la implicación de seguridad se da en el proceso mismo de resolución.
- **Humanas**: cuando la implicación de seguridad se da en una persona o usuario.

Cuando una persona malintencionada encuentra alguna de estas vulnerabilidades, puede tomar acción para **explotarla** y convertirlo en un evento preocupante.

## OWASP

### ¿Qué es OWASP?

**OWASP** (Open Web Application Security Project) es un proyecto de código abierto internacional, sin fines de lucro, que brinda información referente a la seguridad general de aplicaciones web.

OWASP provee un conjunto de herramientas que permiten conocer a profundidad conceptos y herramientas de seguridad:

- **OWASP ZAP**: Proxy para hacer testing de peticiones entre el navegador y aplicaciones.
- **OWASP Juice Shop**: Aplicativo de prueba con vulnerabilidades, donde existen retos de búsqueda y explotación.
- **OWASP Testing Guide**: Guía sobre cómo testear tu aplicación web, con un checklist de vulnerabilidades.

---

### OWASP Top 10

Es un documento que muestra las **10 vulnerabilidades principales** y de impacto más crítico en las aplicaciones web a lo largo de un determinado tiempo. Se actualiza cada 3-4 años.

---

#### A01:2021 – Broken Access Control (Control de acceso roto)

Se produce cuando se logra acceder a un recurso al cual no se debería tener acceso.

**Aplica cuando:**

- Se fuerza una URL para realizar búsquedas a otros recursos.
- Un usuario puede seguir navegando aún cuando su token JWT haya expirado.
- Ruptura de una cookie con un token para elevar privilegios.

---

#### A02:2021 – Cryptographic Failures (Fallos criptográficos)

Uso descuidado de procesos criptográficos, ya sea por aplicación errónea o por algoritmos obsoletos.

**Aplica cuando:**

- Se utiliza algún algoritmo de cifrado obsoleto.
- El proceso de hasheo es intervenido directamente por el desarrollador, generando claves poco sólidas.
- La librería externa que utilizamos no se actualiza correctamente (relacionado con A06).

---

#### A03:2021 – Injection (Inyección)

Inserción de información por parte del usuario con el fin de romper alguna consulta u obtener información sensible.

**Aplica cuando:**

- La data del usuario no fue validada correctamente (ej. `req.body` sin destructurar o validar).
- Un query dinámico sin validación contextual.
- Parámetros malintencionados para realizar consultas a la base de datos (SQL, NoSQL, etc.).

---

#### A04:2021 – Insecure Design (Diseño inseguro)

Conjunto de malas prácticas al momento de construir un aplicativo, desde pequeños detalles hasta elementos estructurales complejos.

**Aplica cuando:**

- Se deja un campo de password en texto plano, sin ocultar.
- Se permite agregar stock con valor `-100`.
- No se aplican patrones de diseño correctamente, generando fuerte acoplamiento.

---

#### A05:2021 – Security Misconfiguration (Malas configuraciones de seguridad)

Descuidos al configurar el aplicativo, módulos o servicios en la nube.

**Aplica cuando:**

- Configuramos erróneamente elementos de arranque (variables de entorno, argumentos).
- Implementamos permisos o características desmedidas (ej. cluster de MongoDB Atlas con IP `0.0.0.0/0`).
- Configuramos mal un cloud service (bucket de AWS S3, ruta de multer).
- Mostramos el stack trace al usuario por mal manejo de errores.

---

#### A06:2021 – Vulnerable and Outdated Components (Componentes vulnerables y obsoletos)

Un componente (externo o propio) se vuelve obsoleto o mantiene una vulnerabilidad no subsanada.

**Aplica cuando:**

- No actualizamos librerías con vulnerabilidades reportadas.
- Dejamos módulos con documentación obsoleta o features sin implementar.
- Actualizamos un módulo sin probar si resolvía la vulnerabilidad inicial.

---

#### A07:2021 – Identification and Authentication Failures (Fallos en identificación y autenticación)

Problemas en el proceso de autenticación que permiten forzar accesos o acceder a información mal tratada.

**Aplica cuando:**

- El sistema de login permite rupturas por fuerza bruta.
- Guardamos contraseñas sin hashear.
- El sistema de “contraseña olvidada” recuerda la contraseña en lugar de restablecerla (la página tenía conocimiento de la contraseña todo el tiempo).

---

#### A08:2021 – Software and Data Integrity Failures (Fallos de integridad de software y datos)

Uso desmedido de módulos, librerías o integraciones de fuentes no confiables, o que presentan alguna vulnerabilidad.

**Aplica cuando:**

- Una dependencia o integración presenta una falla, dejándonos vulnerables también.
- Dependencias que actualizamos sin revisar correctamente (caso Faker.js).
- Procesos CI/CD vulnerados (ej. Heroku + Github con vulnerabilidad de Travis-CI).

---

#### A09:2021 – Logging and Monitoring Failures (Fallos de registro y monitoreo)

No tenemos una correcta gestión de logs y monitoreo del aplicativo.

**Aplica cuando:**

- No hay sistema de monitoreo de APIs para rastrear actividad sospechosa.
- Logs demasiado genéricos que no dan información suficiente para rastrear un error.
- Logs demasiado explícitos que muestran stack trace incluso en producción.
- Logs solo locales sin posibilidad de exportación o filtrado.

---

#### A10:2021 – SSRF (Server Side Request Forgery)

La aplicación no valida la URL que envía un usuario al acceder a un recurso remoto.

**Aplica cuando:**

- El servidor no tiene control de redirecciones o restricción de URL por expresión regular.
- Existe un sistema interno de callbacks que puede desembocar en redirecciones a recursos adicionales.

---

### ¿Qué hago con esta información?

El mundo de la seguridad es mucho más amplio que este Top 10. Sin embargo, como desarrolladores podemos aportar un **grano de arena** cultivándonos en buenas prácticas e implementándolas en nuestros aplicativos.

Para practicar y mejorar tu expertise en seguridad de aplicaciones web, puedes utilizar los proyectos gratuitos de OWASP.

## Documentación

### ¿Qué es documentar?

**Documentar** significa brindar suficiente información sobre algún proceso, con el fin de que sea lo suficientemente entendible para quien lo lea.

La documentación puede darse a nivel simple (comentarios en el código) o a nivel más complejo (herramientas de documentación para un aplicativo).

### ¿Por qué es importante documentar?

**Caso contextual:**  
Trabajas en una empresa de venta de muebles bajo pedido. Solo hay dos desarrolladores: Mauricio (encargado de gestión de muebles y compras) y tú (usuarios, CRM, CMS).  
Mauricio renuncia y te quedas con un código que no hiciste y del que no tienes conocimiento.  
Te solicitan modificar el proceso de órdenes de venta para agregar órdenes premium.

**Problemas al enfrentarte al código:**

- No hay comentarios que guíen.
- No hay recursos que expliquen el flujo o puntos críticos.
- No hay ejemplos de input de `salesOrders` ni `purchaseOrders`.

**Conclusión:**  
Un código no documentado es una bomba de tiempo. Dificulta el mantenimiento, incluso para el autor después de meses.

## Swagger

### ¿Qué es Swagger?

**Swagger** es una herramienta de documentación de código que permite mantener cada módulo de nuestra API dentro de un espectro de entendimiento sólido. Con ella podemos crear nuestra propia **OpenAPI Specification**.

**OpenAPI Specification** (antes conocida como Swagger Specification) es un formato de descripción de REST APIs, escrito en YAML o JSON. Permite profundizar sobre módulos, rutas, esquemas, parámetros, etc.

---

### Proyecto de ejemplo: Adoptme

Adoptme es un proyecto para adopción de mascotas con las siguientes características:

- **Sistema de usuarios**: obtener todos, obtener por id, crear, actualizar, eliminar.
- **Sistema de mascotas**: obtener todas, crear, actualizar, eliminar.
- **Sistema de adopciones**: obtener todas, obtener por id, crear adopción (usuario + mascota).

Puedes clonar el proyecto [aquí]. No maneja entornos; solo reemplazar la URL de MongoDB en `app.js`.

---

### Instalación de Swagger

```bash
npm install swagger-jsdoc swagger-ui-express
```

### Configuración inicial

```js
// swaggerOptions.js
const swaggerJsdoc = require("swagger-jsdoc");

const options = {
  definition: {
    openapi: "3.0.0",
    info: {
      title: "API Adoptme",
      description: "Documentación de la API para adopción de mascotas",
      version: "1.0.0",
    },
  },
  apis: ["./docs/**/*.yaml"], // ruta a los archivos de documentación
};

const swaggerSpec = swaggerJsdoc(options);
module.exports = swaggerSpec;
```

### conectamos Swagger a Express:

```js
const express = require("express");
const swaggerUi = require("swagger-ui-express");
const swaggerSpec = require("./swaggerOptions");

const app = express();

app.use("/api-docs", swaggerUi.serve, swaggerUi.setup(swaggerSpec));

// ... resto de middlewares y rutas
```

#### La documentación mínima debe incluir:

- Schemas
- Routes
- Inputs
- Responses

#### Estructura de carpetas

docs/
users.yaml
pets.yaml
adoptions.yaml

#### Definición de rutas (paths)

```yml
# docs/users.yaml
paths:
  /api/users/:
    get:
      summary: Obtiene todos los usuarios
      tags:
        - Users
      responses:
        "200":
          description: Lista de usuarios obtenida exitosamente
        "500":
          description: Error interno del servidor
```

#### Try it out

Swagger UI incluye un botón Try it out que permite ejecutar la petición directamente desde la interfaz.

#### generando componentes (schemas)

```yml
components:
  schemas:
    User:
      type: object
      properties:
        _id:
          type: string
          description: ID autogenerado de MongoDB
        first_name:
          type: string
          description: Nombre del usuario
        last_name:
          type: string
          description: Apellido del usuario
        email:
          type: string
          description: Correo electrónico
        password:
          type: string
          description: Contraseña (hasheada)
      example:
        _id: "60d21b4667d0d8992e610c85"
        first_name: "Juan"
        last_name: "Perez"
        email: "juan@example.com"
        password: "$2b$10$..."
```

#### Referenciar esquemas en respuestas

```yml
paths:
  /api/users/:
    get:
      responses:
        "200":
          description: Lista de usuarios
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: "#/components/schemas/User"
```

#### Parámetros de ruta

```yml
paths:
  /api/users/{uid}:
    get:
      summary: Obtiene un usuario por ID
      parameters:
        - name: uid
          in: path
          required: true
          description: ID del usuario
          schema:
            type: string
```

#### Definir el req.body (requestBody)

```yml
components:
  requestBodies:
    updateUser:
      description: Objeto con los datos a actualizar
      required: true
      content:
        application/json:
          schema:
            type: object
            properties:
              first_name:
                type: string
              last_name:
                type: string
              email:
                type: string
```

#### Y luego referenciarlo en el endpoint PUT:

```yml
paths:
  /api/users/{uid}:
    put:
      summary: Actualiza un usuario
      parameters:
        - name: uid
          in: path
          required: true
          schema:
            type: string
      requestBody:
        $ref: "#/components/requestBodies/updateUser"
      responses:
        "200":
          description: Usuario actualizado
```
